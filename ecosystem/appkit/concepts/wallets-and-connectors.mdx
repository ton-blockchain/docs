## Wallet

The user's signing application. It holds the keys and signs transactions and data when the user approves.

AppKit collects wallets across every connector into one list — multiple wallets at once is normal — and tracks which one is **selected**: the wallet mutations and signing flows will use.

Every wallet exposes the same minimum surface: a back-reference to the connector that produced it, four identity readers (`getAddress`, `getPublicKey`, `getNetwork`, `getWalletId`), and the two signing operations (`sendTransaction`, `signData`) that flow through to that connector.

```ts
interface WalletInterface {
  /** Connector that created this wallet */
  readonly connectorId: string;

  // Identity
  getAddress(): UserFriendlyAddress;
  getPublicKey(): Hex;
  getNetwork(): Network;
  getWalletId(): string;

  // Actions requiring wallet signature
  sendTransaction(request: TransactionRequest): Promise<SendTransactionResponse>;
  signData(payload: SignDataRequest): Promise<SignDataResponse>;
}
```

Wallet ids are derived from `(network, address)`. The same address on a different network produces a different wallet object — apps that move users between mainnet and testnet should not assume id stability across that boundary.

## Connector

The transport between AppKit and a wallet. It opens the pairing flow, reports which wallets are paired, and routes signing requests through to the wallet.

A connector implements the `Connector` interface. Connector ids are unique — registering a second connector with the same id calls `destroy()` on the old one and replaces it.

```ts
interface Connector {
  readonly id: string;
  readonly type: string;
  readonly metadata: ConnectorMetadata;
  destroy(): void;
  connectWallet(network?: Network): Promise<void>;
  disconnectWallet(): Promise<void>;
  getConnectedWallets(): WalletInterface[];
}
```

The bundled `createTonConnectConnector` covers most apps. It wraps `@tonconnect/ui`, restores prior sessions on first read, and resyncs to AppKit's default network whenever it changes.

## Adding a connector

Both paths accept the same input — a `Connector` instance or a factory `(ctx) => Connector` — and route through the same internal `addConnector` method, so the choice is purely about when registration happens.

### At construction

Pass connectors in the `connectors` field of `AppKitConfig`. The constructor runs `addConnector` for each entry in order.

```ts
import { AppKit, createTonConnectConnector } from '@ton/appkit';

const appKit = new AppKit({
  connectors: [
    createTonConnectConnector({
      tonConnectOptions: { manifestUrl: 'https://example.com/tonconnect-manifest.json' },
    }),
  ],
});
```

### After construction

The `addConnector` action registers a connector against an existing `AppKit` instance and returns an unregister function. Use this when the connector depends on values that aren't ready at construction (e.g. a user-scoped manifest URL) or when the app dynamically swaps connectors.

```ts
import { addConnector, createTonConnectConnector } from '@ton/appkit';

const unregister = addConnector(
  appKit,
  createTonConnectConnector({
    tonConnectOptions: { manifestUrl: 'https://example.com/tonconnect-manifest.json' },
  }),
);

// Later, to remove and destroy the connector:
unregister();
```

## React hooks

`@ton/appkit-react` exposes hooks for the main wallet and connector actions and state readers: `useConnect`, `useDisconnect`, `useConnectors`, `useConnectedWallets`, `useSelectedWallet`, and `useAddress`.

A typical connect-or-disconnect button reads the selected wallet and routes to the matching mutation:

```tsx
import { useConnect, useDisconnect, useSelectedWallet } from '@ton/appkit-react';

function ConnectButton() {
  const [wallet] = useSelectedWallet();
  const { mutate: connect, isPending: isConnecting, error } = useConnect();
  const { mutate: disconnect, isPending: isDisconnecting } = useDisconnect();

  if (wallet) {
    return (
      <button
        onClick={() => disconnect({ connectorId: wallet.connectorId })}
        disabled={isDisconnecting}
      >
        {isDisconnecting ? 'Disconnecting...' : 'Disconnect'}
      </button>
    );
  }

  return (
    <>
      <button onClick={() => connect({ connectorId: 'tonconnect' })} disabled={isConnecting}>
        {isConnecting ? 'Connecting...' : 'Connect Wallet'}
      </button>
      {error && <div>{error.message}</div>}
    </>
  );
}
```

Mutation hooks return TanStack Query's `{ mutate, mutateAsync, isPending, error }`. Async data hooks such as `useBalance` return TanStack Query results. Wallet and connector state hooks read AppKit's local store directly: `useSelectedWallet()` returns `[wallet, setWalletId]`, `useConnectedWallets()` returns `WalletInterface[]`, `useConnectors()` returns `Connector[]`, and `useConnectorById(id)` returns `Connector | undefined`. Watch hooks such as `useWatchBalance` return `void` and keep the matching query cache fresh while mounted.

The full hook surface lives next to the source — see [`@ton/appkit-react/docs/hooks.md`](https://github.com/ton-connect/kit/tree/main/packages/appkit-react/docs) for every hook and example, and [`@ton/appkit/docs/actions.md`](https://github.com/ton-connect/kit/tree/main/packages/appkit/docs) for the underlying action each hook wraps.

## Rules

- Do not call mutations or signing actions while `getSelectedWallet` is `null`. The action will throw.
- Do not assume every connector supports every flow. Catch rejections from the wallet and surface a recovery option.
- Do not pin a stale `selectedWalletId` in app state. The value can be reset by the connector at any moment (disconnect, network change, restore).
- In application code, use the public actions or React hooks instead of calling connector methods directly, so AppKit's event flow stays consistent. Connector methods are still the public surface for custom connector implementations.
