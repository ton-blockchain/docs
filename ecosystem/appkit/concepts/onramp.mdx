OnRamp in AppKit covers two flows for getting value into a TON wallet: fiat-to-crypto onramp through external providers and crypto-to-crypto onramp through cross-chain aggregators. Both are surfaced as managers with the same provider pattern.

## Two flows

| Flow           | Manager                | Provider type           | What AppKit returns                                  |
| -------------- | ---------------------- | ----------------------- | ---------------------------------------------------- |
| Fiat onramp    | `OnrampManager`        | `OnrampProvider`        | A redirect URL to a hosted payment page.             |
| Crypto onramp  | `CryptoOnrampManager`  | `CryptoOnrampProvider`  | Deposit instructions on a source chain plus status. |

Bundled fiat providers: Mercuryo, MoonPay, TON Pay. Bundled crypto providers: Layerswap, Swaps.xyz.

## Fiat flow

The user pays in fiat through a hosted page run by the provider.

1. `getQuote` returns price for a fiat amount or a crypto amount.
1. `buildOnrampUrl` returns a URL to the provider's hosted page.
1. The user is redirected and completes payment off-app.

`getQuotes` fans out to all registered providers concurrently and returns successful results, useful for showing best rates.

## Crypto flow

The user deposits another crypto on a source chain and receives funds on TON.

1. `getQuote` returns the conversion rate and fees.
1. `createDeposit` returns deposit instructions, including the source-chain deposit address and any required memo.
1. Poll `getStatus` to track progress through the bridge.

AppKit does not move funds between chains. The provider does.

## Boundaries

AppKit is the integration surface, not the regulated payment processor. KYC, refund windows, and fees are owned by each provider.

## Rule

OnRamp is asynchronous and external. Treat the returned URL or deposit as the start of a long-running off-app process. Verify completion through provider status, not through wallet state.
