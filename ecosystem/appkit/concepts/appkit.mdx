AppKit is the application layer for TON wallet interaction. It owns the wallet connection state, transaction and signing requests, asset helpers, and DeFi provider integration that a web app needs to talk to TON.

The framework-independent core lives in `@ton/appkit`. React bindings live in `@ton/appkit-react` and wrap the same actions with hooks.

## What it owns

A single `AppKit` instance holds four kinds of state: the configured **networks** (mainnet, testnet, and the API client used to read each one), the registered **connectors** that move signing requests to wallets, the **connected wallets** themselves with one of them marked as selected, and the **managers** for swap, staking, and streaming flows. Anything that touches wallet state should go through this one instance.

## Configuration

`AppKit` is constructed once with an `AppKitConfig`. All fields are optional — pass only what your app needs.

```ts
import { AppKit, Network } from '@ton/appkit';
import { createTonConnectConnector } from '@ton/appkit/tonconnect';

const appKit = new AppKit({
  networks: {
    [Network.mainnet().chainId]: { url: 'https://toncenter.com/api/v2/jsonRPC' },
  },
  defaultNetwork: Network.mainnet(),
  connectors: [
    createTonConnectConnector({
      tonConnectOptions: { manifestUrl: 'https://example.com/tonconnect-manifest.json' },
    }),
  ],
});
```

`defaultNetwork` is the network connectors enforce when connecting a new wallet. Leave it `undefined` to allow any network. `ssr: true` enables a server-side-safe construction path for Next.js and similar frameworks.

## Lifecycle

```text
configure → register connectors → restore session → connect wallet → perform actions → observe state → disconnect
```

Connectors that support session restore (TonConnect) reconnect previously paired wallets the first time you read state, so a returning user usually sees their wallet without an extra connect step. After that, `connect` opens the connector flow, mutating actions read the selected wallet and dispatch through its connector, and watchers stream state changes to your UI. AppKit auto-selects the first wallet whenever the connected list changes and the previous selection is gone.

## Action shapes

AppKit exposes its capabilities as standalone functions, not methods on the instance. Every action takes the `AppKit` instance as its first argument, which keeps the surface tree-shakable and testable.

```ts
import { getSelectedWallet, transferTon } from '@ton/appkit';

const wallet = getSelectedWallet(appKit);
if (wallet) {
  await transferTon(appKit, { recipientAddress: 'EQ...', amount: '1.25' });
}
```

There are three shapes. **Queries** read state synchronously or with a one-shot async call (`getSelectedWallet`, `getBalance`). **Watchers** subscribe to changes and return an unsubscribe function (`watchSelectedWallet`, `watchBalance`). **Mutations** trigger a side effect and return a promise (`connect`, `transferTon`, `sendTransaction`).

## React integration

`@ton/appkit-react` wraps every action and watcher in a hook. Mount once with `AppKitProvider`, then call hooks anywhere below.

```tsx
import { AppKitProvider } from '@ton/appkit-react';
import '@ton/appkit-react/styles.css';

function App() {
  return (
    <AppKitProvider appKit={appKit}>
      <Routes />
    </AppKitProvider>
  );
}
```

Hooks come in three matching shapes. Query hooks return TanStack Query's `{ data, isLoading, error, refetch }`. Watch hooks return `void` and keep the matching query cache fresh while mounted. Mutation hooks return TanStack Query's `{ mutate, mutateAsync, isPending, error }`.

## Rules

- Create one AppKit instance for one app runtime. Do not share an instance across windows or processes.
- Pass the instance to actions directly, or provide it to React with `AppKitProvider`. Do not call actions before the instance is constructed.
- Treat connectors and providers as singletons by `id`. Adding one with an existing `id` replaces and destroys the previous.
- Do not call signing or transfer mutations without first checking that a wallet is selected.
