Toncoin is the native asset on TON. AppKit covers reading balances, building transfer requests, sending them through the connected wallet, and tracking settlement. All actions take the `AppKit` instance as the first argument.

Amounts move through these actions as decimal strings in human-readable units. Toncoin uses nine decimal places, so floating-point math will lose precision — keep amounts as strings until you display them.

## Read balance

`getBalance` reads the selected wallet once. `watchBalance` subscribes to changes and returns an unsubscribe function. Both have `*ByAddress` siblings for reading any address rather than the selected one.

```ts
import { getBalance, watchBalance } from '@ton/appkit';

const ton = await getBalance(appKit);
// "12.345678901"

const stop = watchBalance(appKit, {
  onChange: (update) => {
    // update.balance (formatted string), update.rawBalance (nanos), update.address
  },
});
// Call stop() when done.
```

## Build and send a transfer

A transfer is a two-step flow: build a `TransactionRequest`, then pass it to `sendTransaction`. The two steps are exposed because some apps want to inspect or modify the request before signing — for example, to attach a custom payload.

```ts
import { createTransferTonTransaction, sendTransaction } from '@ton/appkit';

const request = createTransferTonTransaction(appKit, {
  recipientAddress: 'EQ...',
  amount: '1.25',
  comment: 'Order #1001',
});
const response = await sendTransaction(appKit, request);
// response.boc, response.normalizedHash
```

| Parameter           | Required | Meaning                                                                       |
| ------------------- | -------- | ----------------------------------------------------------------------------- |
| `recipientAddress`  | Yes      | Destination address.                                                          |
| `amount`            | Yes      | Amount in TON, as a decimal string. Converted to nanos internally.            |
| `comment`           | No       | Free-text comment. Encoded as a text payload.                                 |
| `payload`           | No       | Base64 BoC. When present, takes priority over `comment`.                      |
| `stateInit`         | No       | Base64 state init for deploying the destination contract on the same message. |

When you do not need to inspect the request, `transferTon` chains build and send into one call:

```ts
import { transferTon } from '@ton/appkit';

const response = await transferTon(appKit, {
  recipientAddress: 'EQ...',
  amount: '1.25',
});
```

Both `createTransferTonTransaction` and `transferTon` throw when no wallet is selected.

## Track settlement

A wallet returning a `boc` only means the user signed. The transaction still has to land on chain. `getTransactionStatus` is a one-shot trace lookup; `watchTransactions` subscribes to the selected wallet's transaction stream.

```ts
import { getTransactionStatus } from '@ton/appkit';

const status = await getTransactionStatus(appKit, { boc: response.boc });
if (status.status === 'completed') {
  // safe to deliver value
}
```

`status` is `'unknown' | 'pending' | 'completed' | 'failed'`. The response also includes `totalMessages`, `pendingMessages`, and `onchainMessages` for trace progress. Pass either `boc` (recommended — the action computes the normalized hash) or `normalizedHash` directly.

## Rules

- Validate the recipient address, amount, and expected network before calling any mutation. Wallets should not be the input validator of last resort.
- Treat `sendTransaction` returning a `boc` as "user signed", not "settled". Wait for `getTransactionStatus` to return `completed` before delivering value.
- Refetch wallet balances and any cached lists after settlement. The wallet response is not the trigger for product state — the trace status is.
- Do not derive product success from the wallet response shape. Verify on the chain.
- Keep amounts as strings until you display them. Floating-point math loses precision at TON's nine decimal places.
