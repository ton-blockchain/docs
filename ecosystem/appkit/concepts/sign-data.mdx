Sign Data lets a wallet attest to arbitrary application data without sending a transaction. Typical uses are user-facing attestations (terms acknowledgments, off-chain receipts) and signatures that a TON smart contract or your backend will later verify. For wallet-ownership sign-in, use `ton_proof` instead — that is a separate flow handled at connection time.

The wallet returns an Ed25519 signature, the signing address, the timestamp it used, and the dApp domain it observed. Verifying that response is the consumer's responsibility.

## Payload kinds

AppKit exposes three payload kinds through three matching mutations and three matching React hooks. Pick the kind that matches what the verifier expects:

| Kind     | Action       | Use                                                                              |
| -------- | ------------ | -------------------------------------------------------------------------------- |
| `text`   | `signText`   | Human-readable strings: sign-in messages, terms of service.                      |
| `binary` | `signBinary` | Arbitrary bytes the wallet should not interpret. Encoded as base64.              |
| `cell`   | `signCell`   | A TON cell with a TL-B schema. Designed for on-chain verification.               |

All three return the same response shape:

```ts
interface SignDataResponse {
  signature: string;     // base64 Ed25519 signature
  address: string;       // signing wallet address
  timestamp: number;     // wallet-supplied unix timestamp
  domain: string;        // dApp domain observed by the wallet
  payload: SignDataRequest; // echo of the original request
}
```

Each sign action below takes an optional `network` to pin the signing network and throws when no wallet is selected.

## Sign text

`signText` signs a UTF-8 string. The wallet receives the literal text and displays it to the user verbatim before asking them to confirm — per the TonConnect spec, this is the format to use whenever the data is human-readable text.

Use it for user-facing attestations the user is meant to read on the wallet screen: terms-of-service acknowledgments, off-chain receipts, agreements you record server-side. **Do not** use it for wallet-ownership sign-in — that is `ton_proof`. Avoid embedding secrets or anything you do not want a user to see — the text is visible.

```ts
import { signText } from '@ton/appkit';

const response = await signText(appKit, {
  text: 'Sign in to example.com\nnonce=2c8e9...',
});
```

## Sign binary

`signBinary` signs an opaque blob, passed as a base64 string. The wallet does not parse the bytes — most wallets render a hash for the user to confirm — so the user cannot meaningfully read what they are signing.

Reach for it when neither text nor cell fits: compact binary protocols, hash chains, or signing the digest of a larger off-chain document. Also a safe choice when you need byte-for-byte stability and worry that a text encoding step might mutate the payload before signing.

```ts
import { signBinary } from '@ton/appkit';

const response = await signBinary(appKit, {
  bytes: 'iJM7Wb...=', // base64 of the bytes you want signed
});
```

## Sign cell

`signCell` signs a TON cell encoded as a Bag-of-Cells (BoC) plus a TL-B schema describing the cell layout. The wallet uses the schema to parse the cell, render the fields to the user, and compute the canonical signing input. The verifier — your backend or a smart contract — uses the same schema to reconstruct that input.

The spec calls for cell in two cases: when the data is destined for the TON blockchain (i.e., a contract will verify the signature), and when the data is structured but not plain text and should still be rendered to the user. Cell is the only kind a TON contract can verify on-chain. The schema is load-bearing in both cases: the wallet relies on it to display fields, and the verifier relies on it to reconstruct the signed input — a layout mismatch means the signature will not check out even if the data is equivalent.

```ts
import { signCell } from '@ton/appkit';

const response = await signCell(appKit, {
  cell: 'te6cck...=', // base64-encoded BoC
  schema: 'auth#_ nonce:uint64 ttl:uint32 = Auth;',
});
```

## Verify a response

Verification is the verifier's job — your backend for off-chain flows, a TON contract for on-chain. The response has to be real, fresh, and meant for this dApp; each field below maps to one check the verifier should make.

| Field       | Check                                                                                          |
| ----------- | ---------------------------------------------------------------------------------------------- |
| `signature` | Verify against the public key of `address` using Ed25519. Signature derives from `payload`.    |
| `address`   | Match against your expected signer (e.g. the connected wallet, an approved key).               |
| `timestamp` | Within an acceptable window (e.g. ±5 minutes). Reject older or future-dated values.            |
| `domain`    | Match the dApp domain you expect. Reject if missing or wrong.                                  |
| `payload`   | Match the request you issued. For text, compare strings. For binary, compare base64 verbatim.  |

For replay protection, embed a nonce that you record server-side and reject on reuse. Wallets do not enforce nonce uniqueness — that is squarely the verifier's job.

## Rules

- Always include a domain identifier and a nonce in the signed payload. Wallets cannot infer either for you.
- Verify the response server-side. Browser-side verification is fine for UX, but settlement must rely on the backend.
- Match `payload` byte-for-byte. Reformatted payloads will fail verification even if the content is equivalent.
- Use `signCell` when a TON contract verifies the signature. Use `signText` for human-readable assertions. Use `signBinary` only when neither fits.
