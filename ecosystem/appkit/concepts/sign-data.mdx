Sign Data lets a wallet attest to arbitrary data without sending a transaction. Use it for sign-in, attestation, and on-chain signatures that a smart contract can verify. The wallet returns an Ed25519 signature, the signing address, the timestamp it used, and the dApp domain it observed. Verifying that response is the consumer's responsibility.

## Payload kinds

AppKit exposes three payload kinds through three matching mutations and three matching React hooks. Pick the kind that matches what the verifier expects:

| Kind     | Action       | Use                                                                              |
| -------- | ------------ | -------------------------------------------------------------------------------- |
| `text`   | `signText`   | Human-readable strings: sign-in messages, terms of service.                      |
| `binary` | `signBinary` | Arbitrary bytes the wallet should not interpret. Encoded as base64.              |
| `cell`   | `signCell`   | A TON cell with a TL-B schema. Designed for on-chain verification.               |

All three return the same response shape:

```ts
interface SignDataResponse {
  signature: string;     // base64 Ed25519 signature
  address: string;       // signing wallet address
  timestamp: number;     // wallet-supplied unix timestamp
  domain: string;        // dApp domain observed by the wallet
  payload: SignDataRequest; // echo of the original request
}
```

## Choosing a kind

`signText` is for human-readable assertions — the wallet typically displays the text in plain form, so avoid embedding secrets or anything you do not want a user to read. `signBinary` is for arbitrary bytes the wallet should not interpret; wallets typically render a hash, so do not assume the user can read what they are signing. `signCell` is the only kind suitable for on-chain verification by a contract — schemas are how the wallet, the verifier, and the contract agree on layout.

```ts
import { signText, signBinary, signCell } from '@ton/appkit';

await signText(appKit, { text: 'Sign in to example.com' });

await signBinary(appKit, { bytes: 'iJM7Wb...=' /* base64 */ });

await signCell(appKit, {
  cell: 'te6cck...=',                  // base64-encoded BoC
  schema: 'auth#_ nonce:uint64 ttl:uint32 = Auth;',
});
```

Each call accepts an optional `network` to pin the signing network. All three throw when no wallet is selected.

## Verify a response

The wallet returns a signature. The consumer (your backend, or another contract) must check that the response is fresh and intended for them.

| Field       | Check                                                                                          |
| ----------- | ---------------------------------------------------------------------------------------------- |
| `signature` | Verify against the public key of `address` using Ed25519. Signature derives from `payload`.    |
| `address`   | Match against your expected signer (e.g. the connected wallet, an approved key).               |
| `timestamp` | Within an acceptable window (e.g. ±5 minutes). Reject older or future-dated values.            |
| `domain`    | Match the dApp domain you expect. Reject if missing or wrong.                                  |
| `payload`   | Match the request you issued. For text, compare strings. For binary, compare base64 verbatim.  |

For replay protection, embed a nonce that you record server-side and reject on reuse. Wallets do not enforce nonce uniqueness — that is squarely the verifier's job.

## Rules

- Always include a domain identifier and a nonce in the signed payload. Wallets cannot infer either for you.
- Verify the response server-side. Browser-side verification is fine for UX, but settlement must rely on the backend.
- Match `payload` byte-for-byte. Reformatted payloads will fail verification even if the content is equivalent.
- Use `signCell` when a TON contract verifies the signature. Use `signText` for human-readable assertions. Use `signBinary` only when neither fits.
