Install AppKit packages into your project. This page covers package selection and bundler setup. Code samples that use AppKit live on later pages.

Installation is a setup step, not a configuration step. After this page you have packages, types, and the default stylesheet on disk; you do not yet have an `AppKit` instance, a connector, or a registered streaming provider. Those are created later, in [Connectors](/ecosystem/appkit/get-started/connectors) and [Providers](/ecosystem/appkit/get-started/providers).

## Pick a path

| Path                                                                  | Use when                                                              |
| --------------------------------------------------------------------- | --------------------------------------------------------------------- |
| [React App](/ecosystem/appkit/get-started/installation/react-app)     | Your app uses React and you want hooks, components, and styles.       |
| [Vanilla JS](/ecosystem/appkit/get-started/installation/vanilla-js)   | Your app does not use React and you call AppKit core actions.         |

Both paths share the core `@ton/appkit` package. The React path adds `@ton/appkit-react` for the provider, hooks, and styles.

## Browser polyfill

AppKit itself does not import `Buffer` directly, but its dependency chain — primarily `@ton/core` and `@ton/crypto` — expects a `Buffer` global in the browser. Modern bundlers (Vite, esbuild, Rollup, Parcel) do not provide one by default, so the polyfill is a one-time setup chore, not a runtime fix. Without it, the first call into `@ton/core` (typically address parsing) throws a `ReferenceError` before your app code runs.

You have two ways to fix this. Pick one.

### Option 1: Manual polyfill (recommended)

This option keeps your bundler configuration untouched, at the cost of controlling import order in the entry file.

Create a `polyfills.ts` file and import it before anything else in your entry file. This keeps the bundler config untouched.

```ts
// src/polyfills.ts
import { Buffer } from 'buffer';

declare global {
  interface Window {
    global: typeof globalThis;
    Buffer: typeof Buffer;
  }
}

window.global = window;
window.Buffer = Buffer;
```

```tsx
// src/main.tsx — first import, before React or AppKit
import './polyfills';

import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import { App } from './app';

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <App />
  </StrictMode>,
);
```

### Option 2: Vite alias

This option moves the work to the bundler at the cost of a config-file edit. Use it when the entry file's import order is out of your control (frameworks that own `main.tsx`, custom build pipelines).

If you cannot run code before module evaluation, alias the polyfill at the bundler level instead.

```ts
// vite.config.ts
import { defineConfig } from 'vite';

export default defineConfig({
  resolve: {
    alias: { buffer: 'buffer/' },
  },
});
```

For webpack and other bundlers, configure the equivalent alias and the `process` and `Buffer` globals.

## Rule

The polyfill must run before any AppKit import is evaluated. Failing this loads `@ton/core` against an undefined `Buffer` and breaks address parsing at module-evaluation time, before any of your code has a chance to handle the error.

## After installation

Continue with one of the following:

- [Connectors](/ecosystem/appkit/get-started/connectors) creates the `AppKit` instance and renders the connect button.
- [Providers](/ecosystem/appkit/get-started/providers) plugs in API clients, streaming, swap, and staking providers.
- [Basic getter hooks](/ecosystem/appkit/get-started/basic-getter-hooks) reads state from React.
- [Sending transactions](/ecosystem/appkit/get-started/sending-transactions) requests wallet operations.
