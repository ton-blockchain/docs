AppKit errors should be handled by class of failure, not only by message text.

| Class          | Source                      | Typical recovery                                                |
| -------------- | --------------------------- | --------------------------------------------------------------- |
| Configuration  | App setup                   | Fix networks, connectors, or providers before showing the flow. |
| Connector      | Wallet transport            | Disconnect and request a new connection.                        |
| User rejection | Wallet UI                   | Keep state unchanged and allow retry.                           |
| Validation     | App input                   | Fix address, amount, payload, or network before opening wallet. |
| Provider       | RPC, swap, staking, indexer | Retry or show degraded state.                                   |
| Settlement     | Chain/indexer               | Continue tracking or ask backend to verify.                     |

## Rule

Do not parse user-facing wallet text for business logic. Use structured result state when available and verify critical outcomes outside the browser.
