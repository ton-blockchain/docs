Sign Data lets a wallet attest to application data without sending a transaction. Typical use cases include user-facing acknowledgments, off-chain receipts, and signatures that a backend or TON smart contract verifies later. For wallet-ownership sign-in, use `ton_proof` instead; it is a separate flow handled during connection.

The wallet returns an Ed25519 signature, the signing address, the wallet-supplied timestamp, and the dApp domain observed by the wallet. Your verifier is responsible for validating the response.

## Payload kinds

AppKit supports three payload kinds, with matching actions and React hooks. Choose the kind your verifier expects:

| Kind     | Action       | Use                                                                              |
| -------- | ------------ | -------------------------------------------------------------------------------- |
| `text`   | `signText`   | Human-readable strings: sign-in messages, terms of service.                      |
| `binary` | `signBinary` | Arbitrary bytes the wallet should not interpret. Encoded as base64.              |
| `cell`   | `signCell`   | A TON cell with a TL-B schema. Designed for on-chain verification.               |

All three return the same response shape:

```ts
interface SignDataResponse {
  signature: string;     // base64 Ed25519 signature
  address: string;       // signing wallet address
  timestamp: number;     // wallet-supplied unix timestamp
  domain: string;        // dApp domain observed by the wallet
  payload: SignDataRequest; // echo of the original request
}
```

Each signing action accepts an optional `network` and throws when no wallet is selected.

## Sign text

`signText` signs a UTF-8 string. The wallet receives the literal text and can display it to the user before asking for confirmation. Use this format when the payload is intended to be human-readable.

Use it for attestations the user should read in the wallet, such as terms acknowledgments, off-chain receipts, or agreements recorded server-side. **Do not** use it for wallet-ownership sign-in; that is what `ton_proof` is for. Do not include secrets or hidden data in text payloads, because the text is visible to the user.

```ts
import { signText } from '@ton/appkit';

const response = await signText(appKit, {
  text: 'Sign in to example.com\nnonce=2c8e9...',
});
```

## Sign binary

`signBinary` signs an opaque blob passed as a base64 string. The wallet does not parse the bytes, and most wallets render a hash for the user to confirm. Use this only when the user does not need to read the payload itself.

Use binary payloads for compact binary protocols, hash chains, or signing the digest of a larger off-chain document. Binary signing is also useful when the payload must remain byte-for-byte stable and a text encoding step could change it.

```ts
import { signBinary } from '@ton/appkit';

const response = await signBinary(appKit, {
  bytes: 'iJM7Wb...=', // base64 of the bytes you want signed
});
```

## Sign cell

`signCell` signs a TON cell encoded as a Bag-of-Cells (BoC), together with a TL-B schema describing the cell layout. The wallet uses the schema to parse the cell, render fields to the user, and compute the canonical signing input. The verifier, whether backend or smart contract, uses the same schema to reconstruct that input.

Use cell payloads when a TON contract will verify the signature, or when the data is structured, not plain text, and should still be displayed in a meaningful form. Cell is the only payload kind designed for on-chain verification. The schema is part of the signed-data contract: the wallet uses it for display, and the verifier uses it to reconstruct the signed input. If the layout does not match, verification fails even when the underlying values are equivalent.

```ts
import { signCell } from '@ton/appkit';

const response = await signCell(appKit, {
  cell: 'te6cck...=', // base64-encoded BoC
  schema: 'auth#_ nonce:uint64 ttl:uint32 = Auth;',
});
```

## Verify a response

Verification belongs to the verifier: your backend for off-chain flows, or a TON contract for on-chain flows. The response must be authentic, fresh, and intended for your dApp. Each response field maps to a concrete check:

| Field       | Check                                                                                          |
| ----------- | ---------------------------------------------------------------------------------------------- |
| `signature` | Verify against the public key of `address` using Ed25519. Signature derives from `payload`.    |
| `address`   | Match against your expected signer (e.g. the connected wallet, an approved key).               |
| `timestamp` | Within an acceptable window (e.g. ±5 minutes). Reject older or future-dated values.            |
| `domain`    | Match the dApp domain you expect. Reject if missing or wrong.                                  |
| `payload`   | Match the request you issued. For text, compare strings. For binary, compare base64 verbatim.  |
| Nonce       | Extract from `payload` and reject if your server has already accepted this nonce.              |

Wallets do not enforce nonce uniqueness. Include a nonce in the payload, record it server-side, and reject any reuse.

## Rules

- Include a domain identifier in the signed payload when your verifier expects one, and always include a nonce. AppKit does not add these application-level fields for you.
- Verify the response server-side for off-chain flows. Browser-side verification can improve UX, but it should not be the authority for authentication or product state.
- Match `payload` byte-for-byte. Reformatted payloads will fail verification even if the content is equivalent.
- Use `signCell` when a TON contract verifies the signature. Use `signText` for human-readable assertions. Use `signBinary` only when neither fits.
