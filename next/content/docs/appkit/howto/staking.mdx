Staking in AppKit is a provider-backed way to quote stake and unstake intent, then build the matching wallet transaction request.

## Staking model

AppKit coordinates staking through a `StakingManager` and registered staking providers. The manager gives the app one integration surface; the provider owns the details of a pool or protocol.

For liquid staking, the wallet position may appear as a staked-token balance such as tsTON. AppKit can surface that balance and provider metadata, but the position remains wallet-side and protocol-side rather than custody inside AppKit.

## Stake and unstake

A staking flow starts with intent: stake TON into a provider, or unstake an existing position. The provider quotes the operation and builds the transaction request that the selected wallet can approve.

Unstake behavior is provider-specific. Instant withdrawal, delayed redemption, pool liquidity, and routing are not AppKit rules; they belong to the staking provider and the underlying protocol.

## Display data

APY, pool metadata, and redemption information help a developer show the current staking context. They are display data from the provider, not a promise of future yield or settlement timing.

Product state should follow wallet and chain outcomes. A quote can explain the expected operation, while settlement confirms what actually happened.

## Quote and send

Fetch provider metadata for display, quote the stake or unstake intent, build the transaction request, then send it through the selected wallet:

```ts
import { buildStakeTransaction, getStakingProviderInfo, getStakingQuote, sendTransaction } from '@ton/appkit';

const info = await getStakingProviderInfo(appKit, { network });

const quote = await getStakingQuote(appKit, {
  amount: '5',
  direction: 'stake',
  network,
});

const request = await buildStakeTransaction(appKit, {
  quote,
  userAddress: wallet.getAddress().toString(),
});

await sendTransaction(appKit, request);
```

Render APY and pool data from `info`, but use transaction status and refreshed balances to decide whether the position changed.

## Rules

- Show APY, pool metadata, and redemption timing as provider data, not guaranteed outcomes.
- Treat a staking request like any other wallet request: wallet approval is not chain settlement.
- Verify the resulting wallet and chain state before updating product state.
